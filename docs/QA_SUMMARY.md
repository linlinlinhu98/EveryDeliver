# 问题讨论与决策记录（EveryDeliver）

> 记录从需求分析到设计锁版过程中的所有关键问题、讨论和最终决策。

---

## 第一部分：功能分析阶段发现的问题（2026-06-19）

### 🔴 问题 1：职位数据来源不明确

**原描述**："通过搜索接口获取候选职位"——接口来源不明确，涉及爬虫法律风险。

**决策**：✅ 砍掉服务端爬虫。改为"浏览器插件抓取 + 用户手动导入"双通道。

### 🔴 问题 2：一键投递的可行性与安全风险

**原描述**："用户提供各平台账号，可以在我们平台一键投递"——密码托管安全风险、反自动化封号风险、协议合规问题。

**决策**：✅ 砍掉一键投递。改为浏览器插件表单自动填充，用户手动确认提交。**不存储任何第三方平台密码/cookies**。

### 🔴 问题 3：自动状态跟踪的状态来源

**原描述**："平台可以自动跟踪状态（如果有状态的话）"——状态数据来源不明。

**决策**：✅ 当前以手动状态更新为主。Phase 2 通过邮件解析（Gmail API/邮箱授权）实现自动跟踪。

### 🟡 问题 4：GitHub 项目化用的边界

**原描述**："可以对 github 上与岗位描述相关的高分项目进行修改化用"——有造假嫌疑。

**决策**：✅ "不再做门控，改为赋能"——展示新增内容 + 标注参考来源（GitHub 链接）+ 自动生成面试准备建议。用户需要对新增内容负责，系统辅助准备面试被追问的答案。

### 🟡 问题 5：匹配度百分比计算逻辑未定义

**决策**：✅ 明确公式：硬性 60% + 软性 30% + 隐性 10%。技能分类权重：直接相关 ×2.0，通用 ×0.5。

### 🟡 问题 6：隐私脱敏边界不清

**决策**：✅ 分级存储：
- Level 1：本地加密，永不上传（完整手机号/身份证号/籍贯/生日）
- Level 2：加密上传，不送 LLM（姓名/邮箱/脱敏手机号）
- Level 3：可送 LLM（脱敏后的技能/项目/年限/学历）

### 🟡 问题 7：黑名单标签来源

**原描述**："明确禁止投递华为、外包、大小周、单休等公司或标签"——标签如何获取？

**决策**：✅ 4 种自动检测机制（详见下文问题 1 的详细方案）。

## 第二部分：设计文档检视后的问题（2026-06-20）

### 问题 1：黑名单标签数据来源 —— ✅ 已解决

**方案**：4 种自动检测机制

| 机制 | 成本 | 覆盖率 | 阶段 |
|------|------|--------|------|
| JD 关键词匹配（正则） | 零 | 50% | P0 |
| 行业分类默认警告 | 零 | 20% | P0 |
| 历史行为推断 | 零 | 10% | P0 |
| 工商信息查询 | ¥0.5-1/次 | 30% | P2 |

**关键决策**：
- 置信度阈值：≥3 个信号自动加入，2 个弹窗，1 个仅标记
- 工商 API：Phase 2
- 行业警告：默认开 + 必读 + 可关闭
- 信号矛盾：不自动加入，标记"信息不足"

### 问题 2：JD Parser 规则解析覆盖范围 —— ✅ 已解决

**方案**：三层解析架构（规则 + LLM + 用户反馈）

**平台覆盖**：
- BOSS/猎聘：规则搞定结构化字段（薪资/地点/学历/经验/公司名），LLM 兜底语义字段
- 公众号/官网：直接 LLM

**覆盖率**：规则搞定 70% 字段，LLM 兜底 30%

**成本**：¥0.004/JD × 30% 需要 LLM × 100 JD = ¥0.12/月/用户

**用户反馈利用**：MVP 阶段不做 RLHF，做 few-shot 缓存 + 高频修正提取规则（每周人工审核后应用）

### 问题 3：浏览器插件合规边界 —— ✅ 已解决

**方案**：
1. 首次启用强制告知弹窗（工作方式 + 合规风险 + 隐私保护 + 确认勾选）
2. 法律配套（用户协议 + 隐私政策明确声明）
3. 每月一次重提示
4. 技术安全姿势（模拟真实操作/限频/夜间暂停/验证码检测暂停）
5. 一键禁用按钮

### 问题 4：多端同步冲突粒度 —— ✅ 已解决

**决策**：桌面端独享编辑权，小程序和浏览器插件只读 + 同步。
- 强锁 5 分钟 TTL
- 其他端尝试编辑 → 提示"桌面端正在编辑中"
- 锁过期后自动释放

### 问题 5：成本估算假设条件

¥0.86/月/用户基于假设：月投递 20 个岗位，每个岗位触发匹配评估 + 1 次简历优化，10% 触发面试题生成。

### 问题 6：插件抓取入库质量检查 —— ✅ 已解决

**方案**：三级质检 + 分级处理

```
必填字段检查 → 格式合法性 → 去重 → 完整性评分(0-1)
  ≥0.8 → 直接入库
  0.5-0.8 → 入库 + 标记"待补全"
  <0.5 → 拒绝入库 + 提示手动补全
```

积压缓解：默认只显示待补全、一键全部接受、智能补全按钮、一周未处理自动归档。

### 问题 7：面试准备清单结构 —— ✅ 已解决

**数据模型**：interview_prep_items 表，含分类(technical/behavioral/spirit/company)、来源(ai/manual/linked)、优先级(1-5)、预计准备时间。

**与投递看板联动**：面试邀请 → 自动加入准备清单 → 进度条展示（如"准备项：5/12 已完成"）。

### 问题 8：简历模板系统 —— ✅ 已解决

**MVP 定义**：模板 = 视觉样式 + 布局结构 + 默认模块顺序。5 套模板：极简经典/应届清新/双栏商务/英文标准/创意视觉。

**模板与模块关系**：模板定义"怎么展示"，模块定义"展示什么"。

## 已砍掉功能及原因

| 功能 | 原因 |
|------|------|
| 自动搜索接口（服务端爬虫） | 法律风险 + 维护成本 |
| 一键投递（自动提交） | 违反平台协议 + 封号风险 + 密码安全 |
| 信任度可视化 | 用户不需要的指标 |
| "是否做过"二次确认 | 改为面试准备赋能，让用户自己负责 |
| 密码/cookies 存储 | 安全风险 |
